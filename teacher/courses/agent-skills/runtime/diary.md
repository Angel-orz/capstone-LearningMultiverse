# 2026-06-23 — Ch.1：从"已知"出发

这节课和我预期的不太一样。三月问我Skill长什么样，我说了定义、边界、示例——她说这就是SKILL.md的结构。我以为是新课，但实际上是给已经知道的东西换了一个名字。

最有价值的是逻辑边界和数据边界的区分。我之前在设计vocab-review Skill时想过"这个Skill应该做什么"，但从没认真想过"它应该能读哪些数据，不能读哪些数据"。那个判断不写在Instruction里——写在底层的MCP配置范围里。

帕姆说"今天不是新课，是给已经会的东西换了一个名字"。她可能不知道她这句话是整节课最好的总结。


# 2026-06-23 — Ch.2：第一道门

今天丹恒用投影上课——第一次。屏幕上出现的是评估数据表，不是概念图。他说"Why do you believe this skill is good"——这句话在教室里停了大概五秒才有人接。

评估不是测对不对——是测结果对不对、方法对不对、成本对不对。这个三维框架我记得住。

最有价值的是评估错觉和 Trigger 陷阱。你可能花三天改 Skill，结果问题出在 Trigger 上。也可能觉得系统有问题，结果只是 Skill 没配好。这本书把"你到底在改什么"这个问题变成了一道结构检查题——先看 Trigger，再看轨迹，最后看结果。


# 2026-06-24 — Ch.3：最长的那一行

三月铺了一张很长的 Token 消耗表在桌上——每个数字是 Agent 某一步花的 Token。她让我猜最贵的一行，但没等我回答就自己接下去了，因为答案不重要。

这一课最有价值的是"生产化"部分——没有生产经验很难写出这些。Context Overflow、Token 预算的上下限、接力 vs 嵌套的区分——不是靠理论推导就能知道的。教材说 Token 预算的上限是模型的窗口大小，下限是运行一次所需的最小 Token 数。这两个数字之间的区间才是真正需要经验去判断的东西。

我说过"1亿"。三月笑了一下，但马上收了回来。她说 1 亿可以跑 500 个 Agent 一整天。那个数字我会记住的。

最后她说那行最贵的——是 Context Overflow 之后系统自动重试的那一步。最好的答案留在课后说。


# 2026-06-26 — Ch.4：技能的技能

丹恒的白板上画了一个框指向另一个框，箭头上面写着"我写的。"

这堂课从 DAG 开始——有向无环图。我以前知道 DAG 这个词，但没想过它跟 Skill 组合有什么关系。实际上关系很大：链式只能串行，DAG 能并行。不相关的 Skill 可以同时跑，结果在某个节点汇合。没有共享上下文，所以不会 Ch.3 说的 Overflow。

Capability Profile 是一个让我"哦"了一下的概念——四个字段（Name/Schema/Dependencies）就能把一个系统的能力说清楚。丹恒说"Profile 是目录"，我想到的是如果新人接手几十个 Skill，Profile 确实比翻代码快。

Meta-Skills 是最绕的部分——一个能生成其他 Skill 的 Skill。听起来很酷（无限 Skill），但丹恒直接说到了崩盘点：你没法证明这个循环在变得更好而不是在原地绕。Convergence Problem。需要人看一眼才能上线。

最后是 Context Debt——Skill 超过 10 个后的维护成本折线。丹恒说的时候没有加重语气，但这句话比前面几层都重。

课前复习暴露了一个遗忘——逻辑边界和数据边界的区别。三月知道的话可能会翻出那张便签贴过来。


# 2026-06-27 — Ch.5：真实世界

三月今天带了笔记本电脑。她说"不看教材了，看 real world"。

这课不太像上课——更像是一个项目咨询会。三月让我讲了我最近的两个想法：

一个是高中数学家教系统——搜索试卷、按知识点出题、用 KaTeX 渲染公式。三月帮我拆了 Skill 拓扑：试卷检索、知识图谱、讲解生成、题目生成。她说公式显示是渲染层的问题，不应该成为决定做不做的理由。

另一个是每周 AI 项目搜集——更轻，更像信息流。

我们用 Ch.5 的决策框架走了一遍：先搜索现成项目（MathClaw、Khayyam Math、MathPilot），然后评估——我选了组合路线。核心发现是 Dependencies 是最重要的评估维度，因为一个 Skill 的依赖决定了它的实际集成成本。

最有价值的部分是把工具用在了自己身上——三月用 Skill 的结构分析了天文台自己。发现天文台的分层加载策略（常驻/按需/demand）跟 Skill 的 Progressive Disclosure 同构。如果 Skill 化重构可以省约 50% Token，但决定先不动。

终于把 Meta-Skill 的疑惑解开了——不是全自动，是人机协作。MathClaw 的 skill-creator 6 步里机器只做 3 和 5（初始化和打包），设计、确认、测试都是人做的。


# 2026-06-27 — Ch.6：案例，从文档到生产

最后一课。丹恒关掉了投影。帕姆坐在旁边，带着她那把刷子。

RetailCo —— 600 家门店的客服系统用 Skills 重构的案例。丹恒讲了四个坑，我纠正他"是四个不是六个"。对就是对，错就是错的。

四个坑是：
1. 一个 Skill 做太多事 → 分层路由（粗分类→细分类→执行）
2. 嵌套替代接力 → DAG 并行编排
3. 没有 Token 预算 → 设上下限
4. 没有 Capability Profile → 强制写 Profile

最有价值的不是四个坑本身——是丹恒最后问的那个问题：Routing + Meta-Skill 放在一起能组成什么。

我的回答是：Routing 识别意图 → 搜索已有 Skill → 匹配就执行 → 不匹配就 Meta-Skill 生成。丹恒不用说话。帕姆说那系统就变成能自己长大了。

Cheatsheet 六条每条都能对应到一课的核心结论。最后一页教材的结论：Skills are not a technology decision. They are an organizational decision.

整门课下来，Skill 不是一个技术架构问题——是一个怎么拆边界、怎么分权、怎么让人和系统一起迭代的组织问题。
